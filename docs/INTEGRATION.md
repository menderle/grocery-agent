# Integrating the Grocery Agent

The agent is plain MCP (Model Context Protocol), so it works with **any MCP-capable
LLM client** — Claude (Code/Desktop/claude.ai), and any other assistant or agent
framework that speaks MCP. There is no Claude-specific code anywhere.

Two surfaces, pick one:

| Surface | What | When |
|---|---|---|
| `grocery-gateway` | **One server with everything**: shopping (search/cart/coupons, proxied texas-grocery-mcp) + checkout/wallet/policy | Recommended for new setups, remote access, and composition |
| `texas-grocery-mcp` + `heb-checkout` | The two servers separately | Fine locally; what `.mcp.json` registers by default |

## Local (stdio) — any MCP client

```json
{
  "mcpServers": {
    "grocery": {
      "command": "/path/to/grocery-agent/.venv/bin/grocery-gateway",
      "env": { "GROCERY_AGENT_HOME": "/path/to/grocery-agent" }
    }
  }
}
```

- **Claude Code / Claude Desktop:** `scripts/install.sh` already writes `.mcp.json` in
  the project root; opening a session in the folder picks it up.
- **Other MCP clients** (agent SDKs, IDE assistants, etc.): point the client's MCP
  config at the same command. Tool descriptions are self-contained; the gateway also
  ships server `instructions` that teach any model the safe operating workflow.

## Remote (HTTP) — phone apps, hosted assistants, other machines

```sh
make serve-http        # gateway on 127.0.0.1:8787, bearer-token auth, /health
```

Expose it with a Cloudflare tunnel (SETUP.md step 6). Then any remote MCP client uses:

- URL: `https://<your-host>/mcp`
- Header: `Authorization: Bearer <MCP_BEARER_TOKEN from .env>`

This is how the Claude phone app connects (claude.ai → custom connector), and equally
how any other hosted LLM with remote-MCP support would.

## As part of a larger personal-assistant agent

The gateway is designed to be a **sub-agent capability**, not an island:

- **MCP-native PA:** mount/register the gateway like any other tool server, ideally
  under a namespace so tools read as `grocery_place_order` next to `calendar_*`,
  `email_*`. With FastMCP, one line:

  ```python
  from fastmcp.server import create_proxy
  assistant.mount(create_proxy("https://<host>/mcp"), namespace="grocery")
  ```

- **Claude Agent SDK / subagent pattern:** give a `grocery-shopper` subagent only these
  tools and the operating prompt in `prompts/system-prompt.md`.

What makes this safe to embed: **the guardrails live in the server, not the prompt.**
Spend limits, approval modes, quiet hours, and order caps are enforced in
`place_order` itself — a confused or compromised orchestrator model cannot exceed
them, and `blocked` responses are not overridable from the client side.

## Feeding the agent (list intake)

`read_grocery_lists` merges every configured source (`config/lists.yaml`):
Apple Notes, Apple Reminders (Siri-fed), a link-shared Google Doc/Sheet, and the inbox
file. The inbox is also reachable as an authenticated HTTP drop-box — useful for Apple
Shortcuts, Zapier/IFTTT webhooks, or **other agents in a PA constellation**:

```sh
curl -X POST https://<host>/list \
  -H "Authorization: Bearer $MCP_BEARER_TOKEN" \
  --data-binary $'oat milk\npaper towels'
```

## Per-user state (what a new person personalizes)

| What | Where | How |
|---|---|---|
| HEB login/session | `~/.texas-grocery-mcp/` + keyring | "authenticate with HEB" on first run |
| Spend limits, autonomy mode, pickup/delivery | `config/policy.yaml` | edit or tell the agent |
| Payment card | HEB wallet via `scripts/add_card.py` → `update_payment_card` | SETUP.md step 2 |
| Staples & brand preferences | `data/staples.json`, `data/preferences.json` | tell the agent over time |
| Secrets | `.env` (generated by installer, gitignored) | automatic |

Nothing user-specific is hardcoded: paths derive from `GROCERY_AGENT_HOME`, the repo
contains no credentials, and `scripts/install.sh` regenerates machine-specific files.
